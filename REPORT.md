# Отчет по проекту: Document Scanner Computer Vision Pipeline

## 1. Общая информация

Проект представляет собой конвейер компьютерного зрения для автоматического сканирования документов. Система принимает на вход фотографию документа, выполняет предварительное улучшение изображения, находит границы документа, исправляет перспективу, очищает результат и рассчитывает метрики качества.

Основная идея проекта - получить из обычной фотографии результат, максимально похожий на скан, и автоматически определить, пригоден ли он для дальнейшего распознавания текста.

## 2. Актуальность задачи

В реальных условиях документы часто оцифровываются не сканером, а камерой телефона. Такой способ удобен, но входные изображения редко бывают идеальными.

Типичные проблемы:

- документ снят под углом;
- лист имеет перспективное искажение;
- присутствуют тени или неравномерное освещение;
- изображение может быть размытым;
- на фото есть шум;
- границы документа могут сливаться с фоном;
- текст может быть недостаточно контрастным для OCR.

Перед распознаванием текста желательно автоматически подготовить изображение: найти документ, выровнять его, перевести в более контрастный вид, убрать шум и оценить качество результата. Именно эту задачу решает данный проект.

## 3. Цель и задачи проекта

Цель проекта - разработать модульный pipeline обработки изображений документов с автоматической оценкой качества результата.

Основные задачи:

1. Реализовать методы улучшения входного изображения.
2. Выполнить сегментацию границ документа.
3. Найти документ на фотографии.
4. Исправить перспективное искажение.
5. Получить бинарную версию документа.
6. Очистить результат от шума и мелких артефактов.
7. Рассчитать метрики качества.
8. Сформировать итоговый вердикт о пригодности результата.
9. Добавить сравнение разных методов обработки.
10. Подготовить несколько вариантов запуска проекта.

## 4. Используемые технологии

В проекте используются:

- **Python** - основной язык разработки;
- **OpenCV** - обработка изображений, поиск границ, контуры, преобразование Хафа, перспективная коррекция, морфология;
- **NumPy** - работа с массивами и численными вычислениями;
- **Matplotlib** - построение сводных визуализаций;
- **Gradio** - веб-интерфейс для интерактивной демонстрации;
- **Jupyter Notebook** - интерактивный сценарий демонстрации и экспериментов.

Зависимости проекта перечислены в файле `requirements.txt`.

## 5. Архитектура решения

Проект построен по принципу последовательного конвейера:

```text
Enhance -> Segment -> Detect -> Clean -> Decide
```

Более детально обработка выглядит так:

```text
Input image
  -> image enhancement
  -> edge detection
  -> document detection
  -> perspective correction
  -> thresholding
  -> morphological cleaning
  -> quality analysis
  -> final decision
```

Каждый этап вынесен в отдельный модуль. Такой подход упрощает разработку, тестирование и расширение проекта: можно заменить один метод обработки, не переписывая весь pipeline.

Центральным модулем является `src/pipeline.py`. Он вызывает остальные компоненты, собирает промежуточные изображения, рассчитывает метрики и возвращает единый результат обработки.

## 7. Описание основных модулей

### 7.1. `pipeline.py`

Файл `pipeline.py` является ядром проекта. В нем определена функция `run_pipeline()`, которая выполняет полный цикл обработки одного изображения.

Основные действия:

1. Загрузка изображения через OpenCV.
2. Сохранение исходного изображения как этапа `original`.
3. Применение метода улучшения изображения.
4. Построение карты границ.
5. Поиск документа выбранной стратегией.
6. Исправление перспективы.
7. Бинаризация документа.
8. Морфологическая очистка.
9. Расчет метрик качества.
10. Формирование итогового решения.
11. Сохранение промежуточных изображений и отчета.

Результат представлен через `PipelineResult`. В нем хранятся:

- статус успешности;
- изображения этапов;
- словарь метрик;
- итоговый quality score;
- финальный grade;
- причина решения;
- использованные методы enhancement и detection;
- текст ошибки, если обработка не удалась.

Также в `pipeline.py` есть функция `run_comparison()`. Она запускает pipeline для всех комбинаций методов улучшения и стратегий детекции.

### 7.2. `enhancement.py`

Модуль отвечает за предварительное улучшение изображения. Это важный этап, потому что качество исходной фотографии напрямую влияет на поиск границ и документа.

Реализованные методы:

- `CLAHE` - локальное повышение контраста;
- `gamma correction` - коррекция яркости;
- `histogram equalization` - выравнивание гистограммы;
- `denoising` - шумоподавление;
- `upscale` - увеличение изображения;
- `none` - режим без улучшения.

Методы собраны в словарь `ENHANCEMENT_METHODS`, поэтому pipeline может выбирать способ улучшения по строковому имени.

По умолчанию используется CLAHE и увеличение изображения в 2 раза. Это помогает сделать границы и текст более заметными.

### 7.3. `detection.py`

Модуль содержит сегментацию границ, поиск документа и перспективную коррекцию.

Функция `detect_edges()` выполняет:

1. перевод изображения в grayscale;
2. Gaussian blur для подавления мелкого шума;
3. расчет медианы яркости;
4. автоматический подбор порогов Canny;
5. Canny edge detection;
6. dilation для усиления и соединения линий.

Стратегии поиска документа:

- **contour detection** - поиск внешних контуров, сортировка по площади и аппроксимация до четырех точек;
- **Hough lines** - поиск прямых линий через преобразование Хафа;
- **edge analysis** - построение нескольких edge maps с разными порогами и объединение результата.

Если выбран режим `auto`, pipeline пробует несколько стратегий последовательно.

После нахождения документа выполняются:

- упорядочивание углов через `order_corners()`;
- перспективная коррекция через `cv2.getPerspectiveTransform()` и `cv2.warpPerspective()`.

В результате документ, снятый под углом, превращается в прямоугольное изображение.

### 7.4. `cleaning.py`

Модуль отвечает за бинаризацию и морфологическую очистку.

Бинаризация реализована в функции `binarize_document()`. Поддерживаются:

- adaptive Gaussian thresholding;
- Otsu thresholding;
- обычный binary threshold.

Основной вариант для pipeline - adaptive Gaussian thresholding, так как он лучше подходит для фотографий с неравномерным освещением.

После бинаризации применяется морфологическая очистка:

- `close` закрывает маленькие разрывы;
- `open` удаляет мелкие шумовые элементы;
- финальный `close` стабилизирует маску.

Финальная последовательность:

```text
close -> open -> close
```

Результатом является очищенная бинарная маска документа.

### 7.5. `decision.py`

Модуль отвечает за оценку качества результата.

Считаются следующие метрики:

- `blur_score` - размытие через дисперсию Лапласиана;
- `contrast_score` - контраст через стандартное отклонение яркости;
- `brightness_min` - минимальная яркость;
- `brightness_max` - максимальная яркость;
- `sharpness` - резкость через Sobel magnitude;
- `text_coverage_pct` - доля черных пикселей на бинарном изображении.

Итоговый quality score начинается со 100 баллов. Далее из него вычитаются штрафы за проблемы качества:

- сильное размытие;
- низкий контраст;
- плохой диапазон яркости;
- слишком маленькое или слишком большое покрытие текста.

Итоговые категории:

- `PASS` - качество от 80%;
- `PASS (with warnings)` - качество от 50% до 79%;
- `FAIL` - качество ниже 50%.

Такой подход позволяет не только получить обработанное изображение, но и автоматически оценить, насколько оно пригодно для дальнейшего OCR.

### 7.6. `visualization.py`

Модуль отвечает за сохранение результатов и визуализацию.

Он создает:

- отдельные изображения этапов;
- `pipeline_summary.png`;
- `decision.txt`;
- `report.txt`;
- `decision_overlay.png`;
- `corner_detection.png`.

`pipeline_summary.png` особенно полезен для анализа, потому что показывает весь путь обработки: от исходного изображения до очищенной маски.

`corner_detection.png` помогает проверить, правильно ли алгоритм нашел углы документа.

### 7.7. `compare.py`

Модуль выполняет сравнение методов.

Он запускает все комбинации:

```text
5 enhancement methods x 4 detection strategies = 20 runs
```

Методы enhancement:

- `clahe`;
- `gamma`;
- `hist_eq`;
- `denoise`;
- `none`.

Стратегии detection:

- `contour`;
- `hough`;
- `edge`;
- `auto`.

После обработки формируется JSON-отчет `comparison_results.json` и выводится таблица результатов. Лучший вариант выбирается по quality score.

Это делает проект не просто демонстрацией одного алгоритма, а инструментом для сравнения подходов.

### 7.8. `config.py`

Файл `config.py` содержит основные настройки проекта:

- список методов enhancement;
- список стратегий detection;
- коэффициент upscale;
- пороги Canny;
- размер морфологического ядра;
- параметры adaptive thresholding;
- пороги качества;
- директорию вывода.

Такой подход позволяет менять параметры без изменения кода отдельных алгоритмов.

### 7.9. `main.py`

`main.py` - основной CLI-интерфейс проекта. Через него можно запускать обработку изображения, выбирать методы, сохранять summary и запускать comparison mode.

### 7.10. `app.py`

`app.py` запускает Gradio-интерфейс. Пользователь может загрузить изображение, выбрать метод улучшения и стратегию детекции, после чего увидеть основные этапы обработки и итоговое решение.

### 7.11. `video.py`

`video.py` применяет pipeline к кадрам видео. Можно обрабатывать не каждый кадр, а, например, каждый десятый, чтобы ускорить выполнение.

Для каждого обработанного кадра сохраняются:

- номер кадра;
- статус успешности;
- quality score;
- итоговый grade.

В конце формируется текстовый summary.

### 7.12. `samples/generate_test.py`

Скрипт генерирует тестовые документы:

- чистые изображения;
- документы с перспективным искажением;
- размытые документы.

Это позволяет дополнительно проверять pipeline на контролируемых примерах.

## 8. Подробное описание алгоритма обработки

### 8.1. Загрузка изображения

Изображение загружается через `cv2.imread()`. Если файл не найден или OpenCV не может его прочитать, pipeline возвращает неуспешный результат с причиной ошибки.

### 8.2. Улучшение изображения

На входном изображении применяется выбранный метод enhancement. По умолчанию используется CLAHE.

Задача этапа - повысить вероятность успешного поиска документа на следующих шагах.

### 8.3. Подготовка рабочего размера

После enhancement изображение может быть уменьшено до рабочей ширины, заданной в `config.py`. Это нужно для ускорения обработки и более стабильной работы алгоритмов.

При этом сохраняется коэффициент масштаба, чтобы найденные координаты углов можно было корректно перенести обратно.

### 8.4. Поиск границ

На изображении строится edge map через Canny. Пороги Canny рассчитываются на основе медианы яркости, что делает метод более адаптивным к разным изображениям.

После Canny применяется dilation, чтобы границы были более связными.

### 8.5. Поиск документа

Pipeline пытается найти четырехугольник документа. Для этого используются контуры, линии Хафа или анализ нескольких edge maps.

Если документ найден, возвращаются четыре угла. Если документ не найден, pipeline завершает работу с вердиктом `FAIL`.

### 8.6. Упорядочивание углов

Четыре найденные точки приводятся к порядку:

```text
top-left
top-right
bottom-right
bottom-left
```

Это необходимо для корректного перспективного преобразования.

### 8.7. Перспективная коррекция

На основе углов строится матрица преобразования, которая переводит документ в прямоугольный вид.

Результат этапа - изображение `detected_document`, то есть выровненный документ.

### 8.8. Бинаризация

Выровненный документ переводится в grayscale и затем в бинарный вид. Основной метод - adaptive Gaussian thresholding.

Бинаризация нужна для подготовки к OCR и для расчета text coverage.

### 8.9. Очистка

После thresholding результат может содержать шум и мелкие дефекты. Морфологическая очистка уменьшает эти проблемы.

На выходе получается `cleaned_mask`.

### 8.10. Оценка качества

Система рассчитывает метрики качества и общий score. Затем формируется один из трех вердиктов:

```text
PASS
PASS (with warnings)
FAIL
```

## 13. Результаты и наблюдения

Проект успешно выполняет полный цикл обработки документа:

1. Загружает изображение.
2. Улучшает контраст и яркость.
3. Выделяет границы.
4. Находит документ.
5. Исправляет перспективу.
6. Бинаризует результат.
7. Очищает маску.
8. Рассчитывает метрики.
9. Формирует итоговый вердикт.

HTML-отчет показывает, что разные изображения требуют разных комбинаций методов. Например, для одних документов лучше работает histogram equalization, для других CLAHE, а для сложных случаев может быть полезна стратегия edge analysis.

Главный практический вывод: универсального метода обработки для всех фотографий нет, поэтому comparison mode является важной частью проекта.
